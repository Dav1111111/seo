## 1. Что строим

**SEO-платформу одного сайта**, которая работает у меня в браузере.

Цель: я должен видеть «что у моего сайта не так и что делать» — конкретно, по моим данным, без воды и общих советов.

Это не «ещё один SEO-сервис как Ahrefs». Это **личный помощник**, который:
1. Читает мои данные из Яндекс Webmaster, Метрики, Search API.
2. Сравнивает с тем, что **я сам** говорю про свой бизнес («я делаю багги-экспедиции в Абхазию»).
3. Выдаёт **конкретные** действия с примерами из моей же базы.
4. Даёт **общаться** — я могу спросить «а почему», «с чего начать», и помощник отвечает на основе моих данных, не выдумывая.

Не фичи которые нужны:
- Не нужен мульти-тенант. Один владелец, может несколько сайтов.
- Не нужен биллинг.
- Не нужен публичный лендинг или маркетинг.
- Не нужны мобильные приложения.

**Один человек: владелец бизнеса.** Не SEO-специалист, не разработчик. Знает свой бизнес лучше всех, не знает про канонические URL и intent-классификацию.

Что он умеет:

Что он НЕ умеет (и не должен):

- Понимать SEO-термины без перевода.
- Терпеть «загрузка…» дольше 10 секунд без объяснений.
- Доверять тексту, который выглядит как «AI-сгенерированный» (общие слова без цифр).

## 3. Архитектурные принципы (НЕПРИКОСНОВЕННЫЕ)

### Принцип 1. Честность данных

> **«Если факта нет в БД — его нет в UI и нет в ответе LLM»**

Никогда не показывай число, которое ты не можешь объяснить SQL-запросом. Никогда не давай совет, который не основан на конкретной строке БД. Если LLM придумала факт — пользователь это увидит и потеряет доверие навсегда.

**Как защищаться:**
- Каждое предложение в UI должно ссылаться на колонку или запрос.
- Все LLM-вызовы используют `tool_use` с JSON-схемой, не свободный текст.
- Каждое LLM-предложение проверяется substring-фильтром против исходных данных. Если evidence не находится в narrative — отбрасывается.
- Никаких «моков» в UI «для красоты». Если данных нет — показывай пустое состояние с честной причиной.

### Принцип 2. Объясняй пустоту

> **«Каждое пустое поле объясняет ПОЧЕМУ оно пустое»**

Если карточка пустая — рядом текст: «Запросы пока не классифицированы — нажми "Классифицировать" в /queries, чтобы я смог ответить».

Никогда: серый прочерк без объяснения. Никогда: «нет данных» без причины.

Это про доверие. Пользователь должен знать что система **знает что не знает**.

### Принцип 3. Не делай за пользователя того, что он не просил

> **«LLM никогда не пишет в БД сама. Только предлагает.»**

Если LLM хочет что-то изменить (профиль бизнеса, фокус, любую настройку) — она возвращает **proposal** (структура), а UI показывает кнопку «Применить». Пользователь жмёт — записывается. Не жмёт — ничего не меняется.

Это **главная** защита от ошибок LLM. Без этого правила — она однажды сотрёт что-то важное.

### Принцип 4. Один LLM-вызов = один tool с JSON-схемой

> **«Никаких "придумай мне SEO-стратегию" в свободном тексте»**

Каждый LLM-вызов:
1. System prompt с правилом «не выдумывай, только из контекста».
2. User message с **конкретными** данными из БД (примеры, числа, цитаты).
3. Tool с JSON-schema — модель обязана заполнить структурированный ответ.
4. После tool_use — **серверная валидация** каждого поля. Если что-то выглядит подозрительно (число вне диапазона, цитата не из текста) — отбрасываем.

Стоимость: ~$0.01-0.10 за вызов на Haiku/Sonnet.

### Принцип 5. Постепенно

> **«Сначала один сайт работает идеально, потом добавляем второй»**

Не делай заранее: мульти-сайт UI, переключатель, фильтры по сайтам, шеринг. Сделай один сайт. Когда он реально работает у меня в браузере — мы добавим второй и увидим что нужно вынести в общее.

Каждый этап = деплой = моя проверка в браузере. Не «закончил всё, теперь смотри». Этапы по 1-3 дня, не больше.

## 4. Технологический стек (фиксированный)

| Слой | Что | Почему |
|---|---|---|
| Backend | **Python 3.12 + FastAPI** | Async, типы, OpenAPI из коробки |
| ORM | **SQLAlchemy 2.0 async** | Стандарт. Не Tortoise, не Django ORM |
| База | **PostgreSQL 16 + JSONB** | JSONB для всех «гибких» полей: target_config, understanding, focus, snapshot |
| Очередь | **Celery + Redis** | Любая фоновая работа (LLM, краулинг, API) идёт через Celery |
| Frontend | **Next.js 16 (App Router) + React 19 + TypeScript strict** | Server components где можно, client где нужен state |
| Стили | **Tailwind 4** | Утилитарные классы. Никаких CSS-in-JS |
| Компоненты | **shadcn/ui** (только Card, Button, Badge, Skeleton, Dialog) | Не тяни большие UI-библиотеки |
| Данные | **SWR** | НЕ React Query. SWR проще |
| LLM | **Anthropic API напрямую** (через `anthropic` SDK) | НЕ LangChain. tool_use из коробки |
| Деплой | **Docker Compose на одном VPS** | Один файл, любой провайдер. nginx → frontend + backend |
| TLS | **Через провайдера** если есть wildcard (Jino так делает) | Не возись с certbot если хостинг сам терминирует |

**Не трогай**: Kubernetes, AWS, GCP, micro services, gRPC, GraphQL, монорепо тулинг (turbo/nx), state-managers (Redux/Zustand/MobX), CSS-in-JS, ORM-генераторы.

Правила:

- Если `ENCRYPTION_KEY` пустой, backend в development может показать понятную инструкцию как сгенерировать ключ, но в production обязан отказаться стартовать.
- `.env.example` обновляется в том же PR, где добавлена новая переменная.
- Перед живыми модулями есть отдельная проверка ключей: Anthropic отвечает, Webmaster видит host, Metrica видит counter, Yandex token не протух.

## 5. Главные структуры данных

### `sites` — корень всего

```sql
CREATE TABLE sites (
    id UUID PRIMARY KEY,
    domain VARCHAR(255) UNIQUE NOT NULL,
    display_name VARCHAR(255),

    -- Подключения к Яндекс
    yandex_webmaster_host_id VARCHAR(255),
    yandex_metrica_counter_id VARCHAR(50),
    yandex_oauth_token TEXT,  -- зашифровано

    -- Где живёт всё гибкое
    target_config JSONB DEFAULT '{}',
    understanding JSONB DEFAULT '{}',

    onboarding_step VARCHAR(50),
    is_active BOOLEAN DEFAULT TRUE,
    created_at TIMESTAMPTZ DEFAULT NOW(),
    updated_at TIMESTAMPTZ DEFAULT NOW()

```

    "products": ["багги-экспедиции"],

{

### Остальные таблицы — нормализованные

| Таблица | Что хранит |
|---|---|
| `pages` | URL, title, h1, meta, content_text, http_status, in_yandex_index, last_crawled_at |
| `search_queries` | query_text, wordstat_volume, relevance (own/adjacent/disputed/spam), is_branded |
| `daily_metrics` | date, metric_type (impressions/clicks/position), dimension_id, value |
| `analysis_events` | site_id, stage, status (started/done/failed/skipped), run_id, message — это «activity feed» |
| `page_reviews` + `page_review_recommendations` | LLM-разбор страницы и пункты «что менять» |
| `coverage_decisions` | strengthen/create/dismiss-решения по парам (intent, page) |
| `outcome_snapshots` | замеры до/после применения рекомендации |
| `chat_conversations` + `chat_messages` | история чата с помощником |

**Правило**: всё что вычисляется из данных — JSONB на site. Всё что нормализуется (страницы, запросы, события) — отдельные таблицы.

## 6. Модули — порядок постройки

Каждый этап = 1-3 дня = деплой = моя проверка в браузере.

### Этап 0: фундамент (полдня)

- Docker Compose: postgres, redis, backend, celery-worker, frontend, nginx
- FastAPI с одним `/health` эндпоинтом
- Next.js с одной страницей `/studio`
- nginx проксирует `/api/*` → backend, остальное → frontend
- Alembic настроен, первая миграция (только sites)
- `.env.example` со всеми переменными из раздела «Переменные окружения и секреты»
- `.gitignore` обязательно содержит `.env`, `.env.local`, dumps и локальные артефакты

**Проверка**: `docker compose up` запускается без ошибок. `/health` отвечает 200. `/studio` показывает «привет».

### Этап 0.5: проверка ключей и живых интеграций (полдня)

До краулера, LLM и Яндекс-модулей убедись, что проект реально видит ключи из `.env`.

Backend:
- config читает `.env` через окружение контейнера;
- endpoint `GET /api/v1/health/config` показывает, какие интеграции сконфигурированы, но **никогда не возвращает сами секреты**;
- endpoint `POST /api/v1/health/integrations/check` запускает короткие проверки Anthropic, Yandex Webmaster и Metrica;
- результат пишется в `analysis_events` со stage `connections_check`.

UI:
- `/studio/connections` показывает Anthropic / Webmaster / Metrica / Search API;
- у каждого сервиса green/yellow/red и простое объяснение: «ключ найден», «токен не работает», «counter id пустой», «нет доступа к host».

Проверки:
- Anthropic: короткий test call выбранной daily-моделью или models/client ping, без дорогого запроса.
- Webmaster: проверить `YANDEX_WEBMASTER_USER_ID` + `YANDEX_WEBMASTER_HOST_ID`.
- Metrica: проверить `YANDEX_METRICA_COUNTER_ID`.

**Проверка владельца**: я кладу реальные ключи в `.env`, запускаю `docker compose up`, открываю `/studio/connections` и вижу зелёные статусы или понятные причины ошибок.

### Этап 1: онбординг сайта (1-2 дня)

UI: форма «Введи домен + OAuth-токен Яндекса». Форма создаёт запись в `sites`.

Backend:
- POST `/sites` создаёт site
- Целевая страница `/studio` показывает домен, кнопку «Прокраулить сайт»

**Проверка**: я ввожу `grandtourspirit.ru`, нажимаю — запись в БД появляется.

### Этап 2: краулер сайта (2 дня)

Celery-таск `crawl_site_task(site_id)`:
1. GET `https://{domain}/robots.txt` → найти sitemap
2. Скачать sitemap.xml, забрать все URL
3. Для каждого URL: GET страницу, спарсить title, h1, meta description, текст
4. Сохранить в `pages`. Активити-event на каждый шаг.

UI: на `/studio/pages` показать список страниц, дату последнего обхода.

**Проверка**: список из 22 страниц grandtourspirit с реальными title.

### Этап 3: понимание бизнеса через LLM (1 день)

Celery-таск `understand_business_task(site_id)`:
1. Берёт первые 10 страниц с самым большим `word_count`
2. Один LLM-вызов (Haiku, tool_use) — возвращает narrative_ru + observed_facts
3. Записывает в `sites.understanding`

UI: страница `/studio/profile` показывает narrative и факты. Кнопка «Перезапустить понимание».

**Защита от выдумок**:
- Tool-схема обязательна
- Каждый observed_fact обязан содержать `page_ref` — реальный URL
- Серверная проверка: page_ref должен быть в `pages.url` для этого сайта

**Проверка**: на grandtourspirit narrative описывает «премиальные багги-экспедиции», observed_facts ссылаются на реальные URL.

### Этап 4: индексация через Webmaster API (1-2 дня)

Celery-таск `webmaster_url_indexation_site_task(site_id)`:
1. `/search-urls/in-search/samples` → список indexed URLs
2. `/search-urls/excluded/samples` → список excluded (soft-fail на 404)
3. Маппит на `pages` (lowercase, no trailing slash)
4. Заполняет `pages.in_yandex_index`, `yandex_excluded_reason`

UI: `/studio/indexation` — таблица «URL → в индексе / исключён / неизвестно».

**Гарантия**: «не знаю» (unknown) **никогда** не показывается как «не в индексе». Это разные состояния.

### Этап 5: запросы из Webmaster (1 день)

Celery-таск `collect_queries_site_task` — забирает запросы из Webmaster API, пишет в `search_queries`.

Wordstat-объёмы — отдельным таском по необходимости (не сразу).

UI: `/studio/queries` — таблица.

### Этап 6: классификатор запросов (2 дня)

`classify_queries_site_task`:
1. Rules-классификатор: точное вхождение primary_product/services + регион → `own`. Иначе `unclassified`.
2. LLM-классификатор для unclassified (Haiku, батчами по 30): own / adjacent / disputed / spam.
3. **Никогда не перетирать `set_by='user'`** — если владелец сам поправил классификацию, LLM это не трогает.

UI: фильтр по relevance, popover «отметить как мой/смежный/спорный/мусор».

### Этап 7: ревью страниц (3 дня)

Этот модуль самый сложный. Работает в три фазы.

**7.1 Decision-tree** (`intent_decide` task):
- Классифицирует запросы по intent (info_dest, comm_category, trans_book — 10 категорий)
- Считает page_intent_scores
- Создаёт coverage_decisions: action ∈ {strengthen, split, leave, create, dismiss}
- Без этого слоя ревью не работает!

**7.2 Reviewer**:
- Принимает (page_id, coverage_decision_id) из решений со strengthen
- ContextBuilder собирает ReviewInput (title, h1, content, top_queries по intent, page profile rules)
- Composite-hash dedup: если контент не менялся, не платим за повторный анализ
- LLM (Haiku/Sonnet) выдаёт PageReview + список Recommendation
- Каждая Recommendation: category, priority, before_text, after_text, reasoning_ru

**7.3 Manual trigger через UI**:
- Кнопка «Запустить ревью» на странице
- Endpoint сначала ищет open strengthen-decision для (site, page_id), если нет — честный skip с понятным сообщением
- Если есть — запускает Reviewer
- 45-сек safety timeout с per-click run-token (чтобы старый таймер не пометил новый run как failed)

**Проверка**: жму «Запустить ревью» на странице с strengthen-decision, через 30 сек вижу 16 рекомендаций.

### Этап 8: модуль конкурентов (2-3 дня)

- Yandex SERP probe для топ-20 запросов сайта
- Сохраняем competitor_profile в target_config
- Deep-dive: краулим топ-5 конкурентов, сравниваем фичи (есть ли формы, цены, schema.org)
- Generate `growth_opportunities` (new_page / strengthen / on_page_feature / schema)

### Этап 9: missing landings (1-2 дня)

LLM-агент:
1. Загружает narrative + observed_facts + список URL сайта
2. Один LLM-вызов: «найди услуги, которые упомянуты в narrative, но нет под них отдельной страницы»
3. **Каждый item обязан иметь evidence_quote** — точная цитата из narrative
4. Серверная проверка: evidence_quote substring-match против narrative. Если не находится — отбрасываем
5. Записываем в `target_config.missing_landings`

### Этап 10: brain — план работ (2 дня)

**НЕ через LLM**. Pure SQL + правила.

- `brain/snapshot.py` — собирает 6 фактов: indexation, queries breakdown, review status, recs pending, missing landings, outcomes
- `brain/rules.py` — 6 правил, каждое читает snapshot и возвращает Action или None
- Action: id, severity, title, body_ru, what_to_do_ru, examples, evidence, in_focus
- Plan = top-N actions, отсортированы (focus, severity)
- Endpoint `GET /sites/{id}/plan` — возвращает Plan
- UI: карточка «Что я бы сделал на твоём месте» сверху на /studio

Severity ladder каждого правила должна иметь min_total guard — на маленьком сайте 4/10 запросов = 40% не должно быть `critical`. Это шум.

### Этап 11: чат с помощником (3 дня)

**11.1 Per-action chat** (на карточках плана):
- Кнопка «Спросить» открывает inline-chat
- LLM получает: action + slice snapshot для action.id + история
- SYSTEM_PROMPT с 5 жёсткими правилами (не выдумывай, не предлагай новые рекомендации, объясняй термины, уважай override, цитируй конкретно)
- Stateless: история в state компонента

**11.2 Free chat** (`/studio/chat`):
- Полный контекст: business profile + understanding + полный snapshot + plan
- Та же защита от выдумок
- **Persistence**: 2 таблицы `chat_conversations` + `chat_messages`. URL `?c=<uuid>` источник правды
- Sidebar list past conversations + delete

**11.3 Tool-use «Применить фокус»**:
- В free chat tool `propose_strategic_focus`
- LLM может либо ответить текстом, либо вызвать tool
- Frontend получает proposal, показывает модалку «Применить?»
- **LLM никогда не пишет напрямую**

### Этап 12: стратегический фокус (1-2 дня)

`target_config.strategic_focus` (см. структуру выше).

UI: блок на `/studio/profile` с chip-полями. Баннер на `/studio` сверху.

**Brain становится focus-aware**:
- Каждое правило проверяет signals (URL/query/service-name) против focus_tokens (substring-match)
- Action.in_focus = bool
- Plan сортируется (focus_bucket, severity, id) — focus-first

**Всё остальное** (free chat system prompt, per-action chat) получает блок «ТЕКУЩИЙ ФОКУС: …».

## 7. Правила работы с LLM (СВЯЩЕННЫЕ)

### Правило 1. Только tool_use со схемой

```python
TOOL = {
    "name": "find_missing_landings",
    "input_schema": {
        "type": "object",
        "properties": {
            "missing": {"type": "array", "items": {...}},
            "summary_ru": {"type": "string"}
        },
        "required": ["missing", "summary_ru"]
    }
}

tool_input, usage = call_with_tool(
    model_tier="cheap",
    system=SYSTEM_PROMPT,  # с правилом «не выдумывай»
    user_message=user_msg,  # с конкретными данными
    tool=TOOL,
)
```

### Правило 2. Каждое поле — пост-валидация

```python
for item in tool_input.get("missing", []):
    # Защита: evidence обязан быть substring источника
    if not evidence_in_signal(item["evidence_quote"], business_signal):
        rejected += 1
        continue  # выбрасываем
    accepted.append(item)
```

### Правило 3. Cost cap на каждый вызов

- Haiku 4.5: $1/1M input, $5/1M output → ~$0.001-0.01 на вызов
- Sonnet 4.6: $3/1M input, $15/1M output → ~$0.005-0.05 на вызов
- Капни `max_tokens` так, чтобы один вызов не превышал $0.10

### Правило 4. Prompt cache на system

```python
system_blocks = [
    {"type": "text", "text": SYSTEM_PROMPT, "cache_control": {"type": "ephemeral"}}
]
```

Это срезает 90% стоимости при повторных вызовах внутри 5-минутного окна.

### Правило 5. Идемпотентность

Любой LLM-таск через Celery должен иметь dedup-окно:
- Краулинг, классификация, не-LLM таски: 60 секунд
- LLM-таски: 600 секунд (10 минут)

```python
recent = await _recent_started_event(
    db, site_id, "missing_landings",
    window_seconds=TRIGGER_DEDUP_WINDOW_LLM_SEC,  # 600
)
if recent:
    return TriggerResponse(status="deduped", run_id=recent.run_id)
```

### Правило 6. LLM никогда не пишет в БД

LLM возвращает `proposal: dict`. UI показывает модалку «Применить?». Только после клика — `POST /apply-proposal` валидирует и пишет.

## 8. UI принципы

### Тон общения

| Плохо | Хорошо |
|---|---|
| «Не классифицировано: 50 запросов» | «Я ещё не разложил запросы по полкам. Запусти "Классифицировать" в /queries.» |
| «Critical: 4 pages have noindex» | «Яндекс не видит 4 твои страницы. Открой раздел «Индексация» — там видно почему по каждой» |
| «No data» | «Ревью на этой странице не запускалось. Жми «Запустить» — займёт 30 сек, ~5 центов LLM.» |
| «Score: 7.2 / 10» | удалить — score без контекста ничего не значит |

### Главное

- **Карточка с действием = одна задача.** Заголовок («что не так»), 2-3 предложения объяснения, конкретные примеры, кнопка «куда нажать».
- **Не показывай 30 цифр на одном экране.** Если их 30 — спрячь под «Показать основание».
- **Цвет = семантика.** Красный = critical, оранжевый = high, жёлтый = medium, зелёный = low / в порядке.
- **Никаких эмодзи в коде.** Иконки = lucide-react SVG. Эмодзи только если пользователь сам их вводит.

### Все clickable элементы

Имеют `cursor: pointer`. Hover-state видно. Focus-ring для клавиатуры.

### Loading states

- Ждёт API: `<Skeleton />`
- Ждёт LLM: «помощник думает…» с пульсирующей искрой
- Ждёт фоновую задачу: показывай прогресс через activity feed (events таблица)

### Безопасность таймеров

Любой `setTimeout` для UI должен:
1. Иметь cancel-token (per-click run-id)
2. Проверять что состояние ещё актуально перед действием
3. Использовать `useTimeoutSetter` хук, который сам убирает на unmount
