> **Статус документа:** контракт. Меняется только согласованно через явное обсуждение и обновление этого файла. PR-ы должны соответствовать тому, что здесь записано — иначе либо PR неправильный, либо документ устарел и его надо обновить **первым**.

## 1. Зачем существует Studio

**Старый интерфейс** (Обзор / Приоритеты / Конкуренты / Отчёты) построен как **витрина модулей**:
кнопка «Запустить полный анализ» дёргает 9 задач параллельно, владелец видит активити-стрим из мелькающих стадий, потом одну общую страницу-комбайн с 8 секциями.

Это **не отвечает** на вопросы, которые реально задаёт владелец сайта:

- Что у меня сейчас в индексе? Если плохо — почему?
- Какие запросы я мог бы взять, но не беру? Где я по ним стою?
- Кто конкретно мои конкуренты? Что у них есть, чего у меня нет?
- Я внёс правку — она сработала или нет?
- Реклама в Директе сливает деньги или окупается?

**Studio** — отдельный раздел продукта, который отвечает на эти вопросы по-человечески.
Каждый вопрос — **отдельный модуль** со своим URL, своим расписанием, своим тестом, своим статусом.
Старый интерфейс **не удаляется** до тех пор, пока Studio не закроет всё функционально.

## 2. Принципы

### 2.1. Один модуль — одна сущность
Каждый модуль отвечает за **один** аспект продвижения. Не «дашборд со всем» — а отдельные страницы:
запросы, индексация, страницы, конкуренты, аналитика, реклама, до/после.
Если модуль начинает делать две вещи — он делится.

### 2.2. Каждый модуль самодостаточен
Любой модуль можно:

- Запустить **отдельно** — кнопкой «Запустить сейчас» или по своему расписанию
- **Перепроверить** — отдельным тестом (через `playground` или unit-тесты модуля)
- **Перезапустить** независимо — упал — починили — снова запустили, остальные модули не затронуты
- Увидеть **здоровье** — green/yellow/red статус, как в `/connectors`

### 2.3. Постоянная работа, не разовая
Модули работают **по графику**, забирая данные **порциями**.

- Wordstat у нас на 22 запроса работает быстро. На 1000 запросов — за один прогон не успеем. Работа идёт пачками: по 50 запросов раз в сутки.
- Запросы пере-проверяются раз в неделю на новую частотность и сезонность.
- Конкуренты пере-проверяются раз в неделю, потому что выдача меняется.
- Метрика собирается ежедневно.
- Расписание у каждого модуля **видно в его UI** + **редактируется** (если нужно временно отключить).

### 2.4. Диагностический workflow
Каждый модуль идёт по одной и той же логике:

1. **Посмотреть** — забрать данные, показать текущее состояние
2. **Диагностировать** — если что-то плохо, найти причину автоматически (как сейчас делает `playground/indexation` — 4 шага проверок и итоговый вердикт)
3. **Предложить действие** — конкретное, не «улучшите title», а «замените «X» на «Y», потому что у конкурента в топ-1 такой формат и слово «цена 2026» добавляет ~2400 показов/мес»
4. **Зафиксировать снапшот** — что было до правки
5. **Применил → отследить эффект** — через 14 дней автоматически собрать новый снапшот и сказать «правка дала +30% показов / -2 позиции / 0% (ничего не поменялось)»

### 2.5. Прозрачность над магией
Никаких чёрных ящиков. Каждый шаг показывает:

- **Что мы отправили** в API (endpoint + body)
- **Что получили** в ответ (структурированно, без сырого JSON по умолчанию)
- **Что это значит** — короткий человеческий вердикт с цветом (зелёный / жёлтый / красный)
- **Сырой JSON** доступен под катом для отладки

Это уже работает в Playground — переносим логику во **все** модули Studio.

### 2.6. Старое не ломаем
Старые маршруты (`/`, `/priorities`, `/competitors`, `/reports`) остаются нетронутыми, пока мы не убедимся что Studio полностью их перекрыла **по функциональности и по доверию**. Когда новый раздел зрелый — старое сносим одним PR.

### 2.7. Документ — раньше кода
Любая значимая правка концепции (добавить модуль, изменить принцип, переразбить раздел) идёт **сначала в этот файл**, потом в код. Иначе документ протухает и становится неактуальным — а нам надо чтобы он оставался живой опорой.

## 3. Каталог модулей

8 модулей. Маршруты под `/studio`. Каждый модуль = отдельная Next.js-страница + (опционально) свой backend-эндпоинт + Celery-задача с расписанием.

| Маршрут | Модуль | Что отвечает |
|---|---|---|
| `/studio/connections` | **Коннекторы** | Подключены ли все Яндекс / Google / Anthropic API. Можно ли им доверять прямо сейчас. |
| `/studio/queries` | **Запросы** | Какие запросы у нас собраны. Объём (Wordstat). Тренд. По каким мы видны, по каким нет. |
| `/studio/indexation` | **Индексация** | Сколько страниц в индексе Яндекса. Если мало — почему (сценарий из Playground). Что чинить. |
| `/studio/pages/[id]` | **Страница** | Workspace одной страницы. Контент, позиции, конкуренты, рекомендации, история правок. |
| `/studio/competitors` | **Конкуренты** | Кто наши реальные конкуренты. Что у них есть, чего нет у нас. |
| `/studio/analytics` | **Аналитика** | Webmaster + Метрика: показы, клики, CTR, поведение, конверсия. |
| `/studio/ads` | **Реклама** | Яндекс.Директ: бюджеты, ROAS. Google Ads (когда добавим). |
| `/studio/outcomes` | **До / После** | История применённых правок и автоматический замер эффекта. |

Подробное описание каждого модуля — в `IMPLEMENTATION.md` по мере разработки. Здесь фиксируем только сам факт существования модуля и его место в иерархии.

## 4. Архитектура: что строим поверх существующего

**Backend.** Большая часть нужной логики уже есть в `app/core_audit/*`:

| Модуль на сервере | Используется в Studio |
|---|---|
| `core_audit/business_truth` | feeds `/queries`, `/competitors` |
| `core_audit/competitors` (discovery + deep_dive + opportunities) | `/competitors` + `/pages/[id]` |
| `core_audit/demand_map` | `/queries` |
| `core_audit/review` | `/pages/[id]` (LLM-рекомендации) |
| `core_audit/priority` | `/pages/[id]` (порядок рекомендаций) |
| `core_audit/report` | будет переосмыслен под Studio-формат |
| `core_audit/outcomes` | `/outcomes` |
| `app/health/connectors` | `/connections` |
| `app/playground/scenarios` | встраивается **в каждый модуль** как «Тестировать пошагово» |
| `collectors/yandex_serp` (Search API + Wordstat) | `/queries`, `/indexation` |
| `collectors/webmaster`, `collectors/metrica` | `/analytics` |
| IndexNow integration (`collectors/indexnow.py`) | `/indexation` |

**Чего нет** на бэкенде и потребует новой работы:

- Yandex.Direct API (бюджеты, кампании, эффективность)
- Google Search Console API
- Google Analytics API
- Расписания на уровне модуля (UI редактируемый cron) — сейчас beat schedule захардкожен в `celery_app.py`

**Frontend.** Полностью новый раздел `/studio/*`. Старое (`/`, `/priorities`, `/competitors`, `/reports`) не трогаем до завершения миграции.

## 5. Анти-цели — чего НЕ делаем

- **Не делаем** комбайны-страницы со всем сразу. Если хочется добавить блок «и тут ещё конкуренты» в `/queries` — это сигнал, что стоит линковать на `/competitors`, а не дублировать.
- **Не показываем** сырой JSON по умолчанию. Только под катом «Показать сырой ответ».
- **Не запускаем** «всё одной кнопкой» как старый «Полный анализ». Каждый модуль самостоятелен.
- **Не удаляем** старый интерфейс до момента, пока в Studio не будут зеркально воспроизведены все ключевые сценарии. Удаляем одним атомарным PR в самом конце.
- **Не объединяем** в один PR несколько модулей. Каждый модуль = свой PR со своими тестами и своим деплоем.
- **Не пишем** «магические» рекомендации без обоснования. Каждый совет должен ссылаться на источник: «у конкурента А в топ-1 это есть, у нас нет», «по Wordstat у фразы X объём Y».

## 6. Метрики успеха раздела

Studio считается успешной, когда выполняются **все** условия:

1. Владелец сайта может ответить на 5 вопросов из §1, **не открывая старый раздел**.
2. Каждый модуль зелёный в `/studio/connections`.
3. У каждой применённой правки есть запись в `/studio/outcomes` с эффектом через 14 дней.
4. Расписания работают — мы не нажимаем «Запустить» руками для рутинных задач, всё крутится по графику.
5. Старый раздел можно снести, и владелец **не заметит** ничего, кроме того, что меню стало чище.

Когда все 5 пунктов закрыты — старый интерфейс удаляется одним PR.
