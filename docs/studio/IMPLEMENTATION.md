## 2. Очередь PR-ов

Каждый PR:
- 1–2 дня работы
- Свои тесты
- Деплой на прод
- Проверка вживую
- Только потом следующий

### 2.1. Скрытые зависимости PR-ов (важно)

- **PR-S4 «фактически становится центром», только когда ниже него уже стоят S2 и S5.** S4 рендерится сам по себе через `core_audit/review`, но cross-link «по этим запросам страница ранжируется → /studio/queries» и «для этой страницы такие конкуренты → /studio/competitors» будут вести в ничего, пока S2/S5 не выехали. Решение: в S4 cross-link'и рендерятся как `<DisabledLink reason="модуль ещё не готов">` пока соответствующий модуль не зеленится.
- **PR-S8 (Outcomes) видит данные только после того, как PR-S4 поработал в проде минимум 14 дней.** Кнопка «Применил» в S4 — это **единственный** триггер записи в `outcome_snapshots`. Без неё /studio/outcomes — пустой бесполезный экран. Поэтому **PR-S8 откладывается до момента «PR-S4 в проде ≥14 дней + есть хотя бы 3 записи в `outcome_snapshots`»**. В очереди оставлен последним намеренно.
- **PR-S3 (Indexation) полностью независим** — играет на playground.indexation, не пересекается с остальным.

### 2.2. Risks of side-by-side old/new UI (учтено)

Plan agent поднял в архитектурном review (журнал §4 ниже). Что закладываем в реализацию **уже с PR-S1**:

- **SWR keys префиксуются `studio:`** — иначе старый `/competitors` и новый `/studio/competitors` шарят кэш и видят чужие мутации.
- **Старые меню-пункты получают `(старый)` суффикс** в sidebar при появлении соответствующих новых модулей. Удаляются только в PR-S9.
- **Idempotency на тригерах модулей.** Если owner кликает «Запустить» в /studio/queries и одновременно дёргает «Полный анализ» из старого `/`, мы не должны платить Wordstat дважды. Реализуется через advisory lock в БД (паттерн `lock_site_target_config` уже есть в `core_audit/sites/locks.py`).

После PR-S8 — пауза на 2-4 недели прод-эксплуатации. Если всё работает — **PR-S9**: атомарное удаление старого раздела (`/`, `/priorities` (старая), `/competitors` (старая), `/reports`).

## 3. Контракт «как добавить модуль»

Каждый новый модуль Studio **обязан** иметь:

### 3.1. Frontend
- Папка `frontend/app/studio/<module>/` с `page.tsx`
- Заголовок (`<h1>`) + краткое описание (1 строка) + breadcrumb
- Карточка «Текущее состояние» — данные из БД, без живых API-вызовов на load
- Карточка «Запустить» — кнопка с осмысленным глаголом, не «Запустить task», а «Найти новые запросы» / «Перепроверить позиции»
- Карточка «Расписание» — когда следующий автозапуск + последний фактический запуск
- Карточка «Здоровье» — индикатор green/yellow/red, как `/studio/connections`
- Карточка «История» — последние 10 запусков с длительностью и результатом
- Кнопка «Тестировать пошагово» — ссылка на playground-сценарий (если применимо)

### 3.2. Backend
- Эндпоинт-триггер `POST /studio/<module>/run` (или переиспользовать существующий из `core_audit/`)
- Эндпоинт чтения `GET /studio/<module>/state` — отдаёт текущие данные + последний запуск + расписание
- Celery-задача с расписанием в `celery_app.beat_schedule` (если модуль работает по графику)
- Health-чек в `app/health/connectors.py` — модуль попадает в общий статус-борд

### 3.3. Тесты
- **Минимум один** unit-тест на ключевую логику модуля
- Если модуль использует внешний API — мок этого API
- Если модуль читает из БД — фикстура на минимум одну запись

### 3.4. Документация
- Описание модуля в `IMPLEMENTATION.md` (этот файл) — что делает, какое расписание, какие зависимости
- Если модуль вводит новый принцип или экзит из `CONCEPT.md` — **сначала** обновить CONCEPT.md, потом писать код

## 4. Журнал технических решений

Сюда записываем **выбранные альтернативы** и **почему**, чтобы через месяц не переоткрывать те же дискуссии.

### 2026-04-25 · Studio как `/studio/*`, не отдельный домен/субдомен
**Выбрано:** один Next.js-app, новый раздел под `/studio`.
**Альтернатива:** отдельный микро-приложение / субдомен.
**Почему:** owner-аутентификация, site context (`useSite`) и API-клиент (`@/lib/api`) уже работают в текущем app. Изоляция через сабдомен добавляет cookies/CORS-проблемы без выгод. Если в будущем понадобится разделить — рефакторинг будет дешевле, чем строить с самого начала.

### 2026-04-25 · Старый интерфейс не удаляется до закрытия Studio
**Выбрано:** старые страницы (`/`, `/priorities`, `/competitors`, `/reports`) живут параллельно со Studio до **PR-S9**.
**Почему:** пользователь явно попросил. Также безопасно: успеем выявить регрессии до того, как удалим точку отката.

### 2026-04-25 · Один PR — один модуль
**Выбрано:** PR-S1 ... PR-S8, каждый отдельно. Не «весь Studio за один спринт».
**Почему:** возможность остановиться после любого PR без потери ценности. Если PR-S3 сломается — у нас есть рабочие S1+S2 в проде, и есть точка чтобы пересмотреть подход.

### 2026-04-25 · Расписания пока остаются в `celery_app.beat_schedule`
**Выбрано:** UI расписаний модуля = read-only в первой итерации. Plus per-site `module_enabled` boolean в БД для возможности «временно отключить модуль на этом сайте».
**Альтернатива:** `django-celery-beat`-style DB-scheduler с динамической перезагрузкой.
**Почему:** редактируемый cron из UI требует динамического перезапуска worker'а или `celery beat --schedule`-приключений (полдня + риск). Boolean enable/disable закрывает 80% потребности §2.3 за пол-дня без архитектурных переделок. Полный редактируемый cron — после PR-S8, если реально захочется.

### 2026-04-25 · PR-S4 откладывает PR-S8 на 14+ дней проду
**Выбрано:** PR-S8 (Outcomes UI) выпускается **не сразу после S7**, а после того как PR-S4 уже **поработал в проде ≥14 дней** и в `outcome_snapshots` есть реальные данные.
**Почему:** «Применил → +14 дней → дельта» — это сама суть Outcomes. Если выпустить S8 пустым, owner откроет один раз, увидит «нет данных», больше не вернётся. Лучше выпустить с уже наполненной историей.

### 2026-04-25 · doc-before-merge с trigger list, не doc-before-code
**Выбрано:** §2.7 CONCEPT.md описывает фиксированный trigger list. PR попадает в список — обязан содержать diff к CONCEPT.md (или явное обоснование «концепция не меняется»). Не попадает — обычный PR.
**Альтернатива:** doc-before-code (предыдущая формулировка) или вообще без правила.
**Почему:** doc-before-code на 2-человечной команде с daily deploy создаёт churn (либо док пишется одновременно с кодом и теряет смысл, либо всё считается «не значимым» и док устаревает). Trigger list убирает спор «значимая ли правка», проверяется механически. Решение поднял Plan agent в architectural review.

## 5. Что НЕ обещаем в Studio v1

Чтобы не разочаровываться:

- **Real-time данные.** Везде есть лаг — Wordstat обновляется раз в неделю, Метрика — раз в час, Search API — синхронный но 5-10 секунд.
- **AI-агент, который сам всё применит.** Все правки требуют ручного «Применил». Так задумано — мы не правим контент клиента без подтверждения.
- **Полная индексация всего сайта.** Wordstat / Search API стоят денег и имеют квоты. Большие сайты обрабатываются порциями по графику.
- **Google интеграция в первой версии.** Только Yandex. Google добавляется отдельным sprint'ом после Studio v1.
- **Multi-tenancy в новом разделе.** Текущий single-site context переиспользуется — Studio для текущего выбранного сайта.
