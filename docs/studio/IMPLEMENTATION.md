## 3. Контракт «как добавить модуль»

Каждый новый модуль Studio **обязан** иметь:

### 3.1. Frontend
- Папка `frontend/app/studio/<module>/` с `page.tsx`
- Заголовок (`<h1>`) + краткое описание (1 строка) + breadcrumb
- Карточка «Текущее состояние» — данные из БД, без живых API-вызовов на load
- Карточка «Запустить» — кнопка с осмысленным глаголом, не «Запустить task», а «Найти новые запросы» / «Перепроверить позиции»
- Карточка «Расписание» — когда следующий автозапуск + последний фактический запуск
- Карточка «Здоровье» — индикатор green/yellow/red, как `/studio/connections`
- Карточка «История» — последние 10 запусков с длительностью и результатом
- Кнопка «Тестировать пошагово» — ссылка на playground-сценарий (если применимо)

### 3.2. Backend
- Эндпоинт-триггер `POST /studio/<module>/run` (или переиспользовать существующий из `core_audit/`)
- Эндпоинт чтения `GET /studio/<module>/state` — отдаёт текущие данные + последний запуск + расписание
- Celery-задача с расписанием в `celery_app.beat_schedule` (если модуль работает по графику)
- Health-чек в `app/health/connectors.py` — модуль попадает в общий статус-борд

### 3.3. Тесты
- **Минимум один** unit-тест на ключевую логику модуля
- Если модуль использует внешний API — мок этого API
- Если модуль читает из БД — фикстура на минимум одну запись

### 3.4. Документация
- Описание модуля в `IMPLEMENTATION.md` (этот файл) — что делает, какое расписание, какие зависимости
- Если модуль вводит новый принцип или экзит из `CONCEPT.md` — **сначала** обновить CONCEPT.md, потом писать код

## 4. Журнал технических решений

Сюда записываем **выбранные альтернативы** и **почему**, чтобы через месяц не переоткрывать те же дискуссии.

### 2026-04-25 · Studio как `/studio/*`, не отдельный домен/субдомен
**Выбрано:** один Next.js-app, новый раздел под `/studio`.
**Альтернатива:** отдельный микро-приложение / субдомен.
**Почему:** owner-аутентификация, site context (`useSite`) и API-клиент (`@/lib/api`) уже работают в текущем app. Изоляция через сабдомен добавляет cookies/CORS-проблемы без выгод. Если в будущем понадобится разделить — рефакторинг будет дешевле, чем строить с самого начала.

### 2026-04-25 · Старый интерфейс не удаляется до закрытия Studio
**Выбрано:** старые страницы (`/`, `/priorities`, `/competitors`, `/reports`) живут параллельно со Studio до **PR-S9**.
**Почему:** пользователь явно попросил. Также безопасно: успеем выявить регрессии до того, как удалим точку отката.

### 2026-04-25 · Один PR — один модуль
**Выбрано:** PR-S1 ... PR-S8, каждый отдельно. Не «весь Studio за один спринт».
**Почему:** возможность остановиться после любого PR без потери ценности. Если PR-S3 сломается — у нас есть рабочие S1+S2 в проде, и есть точка чтобы пересмотреть подход.

### 2026-04-25 · Расписания пока остаются в `celery_app.beat_schedule`
**Выбрано:** UI расписаний модуля = read-only в первой итерации. Изменение через код.
**Почему:** редактируемый cron из UI требует динамического перезапуска worker'а или `celery beat --schedule`-приключений. Это **отдельный модуль** который мы добавим после PR-S8, если реально понадобится.

## 5. Что НЕ обещаем в Studio v1

Чтобы не разочаровываться:

- **Real-time данные.** Везде есть лаг — Wordstat обновляется раз в неделю, Метрика — раз в час, Search API — синхронный но 5-10 секунд.
- **AI-агент, который сам всё применит.** Все правки требуют ручного «Применил». Так задумано — мы не правим контент клиента без подтверждения.
- **Полная индексация всего сайта.** Wordstat / Search API стоят денег и имеют квоты. Большие сайты обрабатываются порциями по графику.
- **Google интеграция в первой версии.** Только Yandex. Google добавляется отдельным sprint'ом после Studio v1.
- **Multi-tenancy в новом разделе.** Текущий single-site context переиспользуется — Studio для текущего выбранного сайта.
