## TL;DR — состояние платформы

**На текущем масштабе (1 владелец, 2 сайта) платформа жизнеспособна.** Есть функциональный пайплайн (14-17 сек wall time), грамотная доменная архитектура (`core_audit/<domain>/`), хорошее покрытие тестами критичных модулей (BusinessTruth 107, demand_map 89, review 47, intent 44 — суммарно ~450 тестов), правильные паттерны (`task_session`, `run_id`, `emit_terminal`, advisory locks).

**Но к мульти-тенантному спринту платформа НЕ готова.** Совпадение трёх независимых аудитов (архитектура, security, tech debt) указывает на одну корневую проблему: **отсутствует фундамент аутентификации и тенант-скоупинга**. Каждая API-ручка `/sites/{id}/...` принимает любой UUID без проверки владельца. `ADMIN_API_KEY` — единственный механизм auth и при этом захардкоден в `docker-compose.yml:93` (`admin_dev_secret_2026`) и закоммичен в git. Включение второго платящего клиента сейчас = немедленный security incident.

**Параллельно есть операционные риски прямо сейчас:** 0 Celery-задач имеют `task_time_limit` (зависшая SERP-задача блокирует воркер навсегда), `reconcile_open_pipelines` пишет в БД на каждом GET activity (3 эндпоинта × poll 5-10s = до 18 reconciles/мин/viewer), connection-pool leak в `agents/tasks.py` и `collectors/tasks.py`, `client.messages.create()` (sync) внутри async FastAPI handler `/chat`.

**Готовность по направлениям:**

| Направление | Оценка | Блокер для prod | Блокер для multi-tenant |
|---|---|---|---|
| Архитектура | B | нет | **ДА** (tenant_id только на Site) |
| Безопасность | D | **ДА** (port-exposure + OAuth plaintext) | **ДА** (нет auth + IDOR на всех ручках) |
| Backend code | B- | **ДА** (Celery time_limit, sync LLM) | нет |
| БД + perf | B | нет (но time_limit горит) | нет |
| Frontend + a11y | C+ | нет | нет |
| Tech debt + deps | B+ | нет | **ДА** (single-tenant хардкод) |

**Источники:** S-003, A-007, C1 tech-debt, A-003.
- `.env` (не в git, но в проде): `ANTHROPIC_API_KEY=sk-ant-***REDACTED***`, `YANDEX_OAUTH_TOKEN=y0__***REDACTED***`, `YANDEX_OAUTH_CLIENT_SECRET=***REDACTED***`, `DB_PASSWORD=devpassword`, `SECRET_KEY=dev-secret-key-change-in-production`.

- `frontend/.env.local` запекается в Docker image через `COPY . .` (нет `.dockerignore`).

**Действие:**
1. Сегодня ротировать: `ANTHROPIC_API_KEY`, `YANDEX_OAUTH_TOKEN`, `YANDEX_OAUTH_CLIENT_SECRET`, `ADMIN_API_KEY`, `DB_PASSWORD`.

4. Проверить git-историю: `git log --all -p | grep -E "sk-ant|y0__|admin_dev_secret"`.
5. На VPS перезапустить Docker compose с новыми env.

#### M-002 — Postgres 5432 и Redis 6379 проброшены на 0.0.0.0
**Источник:** S-005.
- `docker-compose.yml:11-12` — `db.ports: ["5432:5432"]`, пароль `devpassword`.
- `docker-compose.yml:22-23` — `redis.ports: ["6379:6379"]`, **без пароля**.
- VPS на myjino.ru, любой Shodan-скан видит. Redis без пароля исторически даёт RCE через `CONFIG SET dir + SAVE`.

**Действие:** удалить `ports:` блоки у `db` и `redis` (они доступны контейнерам по имени сервиса). Если нужен внешний доступ — `127.0.0.1:5432:5432` + `ssh -L`. Включить `requirepass` для Redis.

#### M-003 — Yandex OAuth токены plaintext в БД (комментарий «encrypted at rest» лжёт)
**Источники:** A-002, S-004, H3 backend, H5 tech-debt.
- `backend/app/models/site.py:19` — `yandex_oauth_token: Text  # encrypted at rest`.
- `ENCRYPTION_KEY` объявлен в `config.py:36`, **не используется ни разу** (`grep encrypt|fernet|cryptography backend/app/` = 0).
- Пакет `cryptography>=44.0.0` уже в `pyproject.toml:20`, но не импортируется.

**Действие:** реализовать `EncryptedString` SQLAlchemy `TypeDecorator` через `cryptography.fernet`, ключ из `ENCRYPTION_KEY`. Миграция: расшифровать существующие 2-3 токена → зашифровать обратно. Удалить лживый комментарий или сделать его правдой.

#### M-004 — 0 Celery-задач имеют `task_time_limit` / `soft_time_limit`
**Источники:** Backend C3, DB CRIT-1.
- `backend/app/workers/celery_app.py:11-45` — конфиг есть, лимита нет.
- `worker_cancel_long_running_tasks_on_connection_loss=False` + отсутствие лимита = зависшая SERP/crawl/deep_dive задача держит воркер часами.
- При `concurrency=2` достаточно одной зависшей задачи на 50% мощности.

**Действие:** в `celery_app.py` добавить `task_soft_time_limit=300` (5 мин), `task_time_limit=420` (7 мин hard SIGKILL). На тяжёлых тасках (SERP, crawl, deep_dive) переопределить декоратором до 600/720.

### P1 — фиксить до конца недели

#### M-005 — Нет аутентификации на пользовательских API-эндпоинтах
**Источники:** S-001, A-007, C1 tech-debt.
- Все роуты `backend/app/api/v1/*.py` кроме `admin_*.py` и `business_truth.py` принимают любой UUID без авторизации.
- `GET /sites`, `GET/PATCH /sites/{id}`, `/dashboard`, `/queries`, `/issues`, `/reports/{id}`, `/priorities/sites/{id}`, `/intent/sites/{id}/...`, `/collectors/...` — всё открыто.
- `PATCH /sites/{id}` (`backend/app/api/v1/sites.py:53`) позволяет переписать `yandex_oauth_token`, `yandex_metrica_counter_id` — full hijack профиля.
- VPS-домен `c6b32ea267b.vps.myjino.ru` явно записан в `backend/app/main.py:25` и утечёт через CORS-заголовки.

**Действие (срочно для single-tenant):** nginx Basic Auth ИЛИ глобальный `Depends(_require_admin)` на `v1_router`. Это не настоящая auth, но хотя бы дверь.
**Перед multi-tenant:** сессии (cookie+CSRF) или Bearer JWT, `Depends(get_current_tenant)`, везде `where(Site.tenant_id == current_tenant.id)`.

#### M-006 — IDOR: `/sites/{site_id}/*` не проверяет владельца
**Источники:** S-002, A-001/A-003.
- Каждый `select(...).where(Site.id == site_id)` без `Site.tenant_id == current_tenant_id`.
- В single-tenant сейчас не страшно. В multi-tenant — каждый клиент видит всех через подбор UUID или утечку из nginx access-log.

**Действие:** ввести `Depends(get_current_site_for_user)` — единственная точка, где смотрят URL `site_id` и проверяют принадлежность tenant'у. Тесты: для каждого роута позитивный + негативный (другой tenant → 404).

#### M-007 — `tenant_id` есть только на `Site`; multi-tenant RLS невозможен
**Источники:** A-001, C2 tech-debt.
- 17+ доменных таблиц (`issues`, `daily_metrics`, `analysis_events`, `search_queries`, `page_reviews`, `target_clusters`, `query_intents`, `page_fingerprints`, `outcome_snapshots`, `tasks`, `pages`, `agent_runs`, `alerts`, `report`, `coverage_decisions`, `target_queries`, `weekly_reports`) несут только `site_id`.
- RLS-политики поверх такой схемы потребуют subquery `EXISTS(SELECT 1 FROM sites WHERE sites.id = THIS.site_id AND sites.tenant_id = current_setting(...))` на каждый row check — медленно и хрупко.

**Действие (multi-tenant спринт, неделя 1):** Alembic-миграция: добавить `tenant_id UUID NOT NULL` (сначала nullable + backfill SQL `UPDATE t SET tenant_id = (SELECT tenant_id FROM sites WHERE sites.id = t.site_id)` + `CREATE INDEX CONCURRENTLY (tenant_id, site_id, …)` + во второй миграции NOT NULL). FK на `tenants.id` в моделях.

#### M-008 — `X-Admin-Key` — единственный auth для всех write-эндпоинтов; не constant-time
**Источники:** A-003, S-101.
- `_require_admin` дублируется в 3 файлах: `admin_ops.py:30`, `admin_demand_map.py:47`, `business_truth.py:22`. Сравнение `if not x_admin_key or x_admin_key != configured` — early-exit, timing-attack возможен.
- Прокси `frontend/app/admin-proxy/[...path]/route.ts:4,21` транзитивно инжектит ключ — backend не знает, какой пользователь вызвал admin-action. В multi-tenant — horizontal escalation тривиален.

**Действие:**
1. Сейчас: `hmac.compare_digest(x_admin_key, configured)` + вынести `_require_admin` в `api/v1/deps.py`.
2. До multi-tenant: заменить shared-secret на JWT/Session с `tenant_id` и `user_id`.
3. На каждом `/admin/sites/{id}/...` — `Depends(require_site_owner(site_id))`.

#### M-009 — `reconcile_open_pipelines` пишет в БД на каждом GET activity
**Источники:** DB CRIT-2, A-014.
- `backend/app/api/v1/activity.py:53,71,104` — каждый из 3 GET-эндпоинтов начинается с `await reconcile_open_pipelines(db, site_id)`.
- `frontend/components/dashboard/activity-feed.tsx:54` — `refreshInterval: 5000`. Плюс `last-run-summary.tsx:50` (4s), `overview.tsx:128` (10s).
- Внутри: `SELECT … LIMIT 50` + N×3 запросов на каждую открытую `pipeline:started` строку. На пустом стейте — 18 reconciles/мин/viewer.
- Нарушает HTTP-семантику (GET = pure read).

**Действие:** вынести reconcile в отдельную Celery beat-таску раз в 30s (можно подсадить на существующий `queue_health_check` каждые 120s). Альтернатива — кеш в Redis с TTL 30s на ключ `reconcile:{site_id}`.

#### M-010 — Connection-pool leak в `agents/tasks.py` и `collectors/tasks.py`
**Источники:** A-004, Backend C1.
- 11 вызовов `_make_session()` (`collectors/tasks.py:94,112,125,193,249,329` + `agents/tasks.py:51,73,117,148,198`) создают `create_async_engine(...)` и **не вызывают `engine.dispose()`**.
- Правильный паттерн уже есть: `backend/app/workers/db_session.py:28-41` (`task_session()` с `await eng.dispose()`).
- При `worker_max_tasks_per_child=50` это до 100 висящих connection-пулов на цикл жизни воркера.

**Действие:** заменить `_make_session()` на `task_session()` во всех 11 местах. 2-4 часа работы. CI-проверка: `grep "_make_session" backend/app/` = 0.

#### M-011 — Sync `client.messages.create()` внутри async FastAPI handler
**Источник:** Backend C2.
- `backend/app/api/v1/chat.py:177` — `response = client.messages.create(...)` (sync) в `async def`.
- `get_client()` в `agents/llm_client.py:57` возвращает `anthropic.Anthropic` (sync), не `AsyncAnthropic`.
- Блокирует event-loop FastAPI на 5-60 секунд каждого LLM-вызова. При одновременных чатах 2-3 пользователей весь dashboard замораживается.

**Действие:** либо отдельный `anthropic.AsyncAnthropic` клиент для async-кода, либо `await asyncio.to_thread(client.messages.create, ...)`. 30 минут.

## Что в проекте сделано хорошо — НЕ ЛОМАТЬ

Это важная секция, чтобы при рефакторинге не выбросить правильное вместе с проблемным.

1. **Domain-package структура `core_audit/<domain>/`** (`business_truth`, `priority`, `report`, `competitors`, `review`, `demand_map`, `intent`, `fingerprint`) с `dto.py` + `service.py` + `tasks.py` + `models.py`. Эталонный DDD-паттерн, легко расширять. Не ломать.

2. **`task_session()` примитив** в `backend/app/workers/db_session.py:28-41`. Документация прямо в docstring объясняет, почему `eng.dispose()` обязателен. Это правильный путь — нужно лишь домигрировать туда `agents/tasks.py` и `collectors/tasks.py`.

3. **`run_id` контракт для группировки событий** (`admin_ops.py:82-111`, `core_audit/activity.py`). Прокидывается через `kwargs` во все task'ы пайплайна. UI отделяет «текущий клик» от «прошлый» — это сильное архитектурное решение.

4. **`emit_terminal` invariant** (`core_audit/activity.py:230-294`): «каждый pipeline:started получает pipeline:<terminal> в течение 10 минут», закреплено тестами в `backend/tests/competitors/`.

5. **Onboarding gate** (`core_audit/onboarding/gate.py`): nightly job берёт только сайты с `onboarding_step="active"`. Manual triggers намеренно bypass'ят — правильный дизайн.

6. **Cloudflare Worker прокси для Anthropic** + streaming в `agents/llm_client.py:108`. Правильное решение проблемы геоблокировки + Vercel timeout. Комментарии объясняют **почему**.

7. **Idempotency через `pg_try_advisory_lock`** в `core_audit/competitors/tasks.py:38-40,242-255`. Хорошая практика для long-running tasks без отдельной очереди.

8. **Healthcheck Celery worker через `inspect ping`** (`docker-compose.yml:62`). Решает реальную проблему «процесс жив, Redis отвалился» — обычные процесс-уровневые проверки её бы пропустили (см. `MEMORY.md` про Celery Redis UNBLOCKED).

9. **Pydantic `model_config=extra=allow` для `TargetConfigBody`** (`admin_demand_map.py:80`) — future-proof JSONB API.

10. **Параметризованный SQL везде** (нет ни одного `text()` с f-string), Pydantic-валидаторы на входе, `expire_on_commit=False` корректно. Базовая security-гигиена в коде на месте.

11. **Frontend admin-key изоляция** (`frontend/app/admin-proxy/[...path]/route.ts:4`) — `ADMIN_API_KEY` живёт только в server-route, в браузер не уходит. Это сделано правильно.

12. **Покрытие тестами BusinessTruth (107) и demand_map (89)** — лучшие тестовые папки в проекте. Эталон того, как покрывать новый модуль.

13. **Celery resilience-config** (`broker_connection_retry_on_startup`, `visibility_timeout=3600`, `worker_cancel_long_running_tasks_on_connection_loss=False`) — видно, что прошли через реальный production-инцидент.

14. **Operating mode guard** (`services/operating_mode.py`) — простая, но честная архитектура `readonly/recommend/propose/autoexecute`. `IntEnum` — extensible.

15. **Migrations: 14 миграций в одну линейную цепочку, 0 split-head, все используют `timestamptz` на уровне БД.** Только модели рассинхронизированы — лечится.

## Финальная оценка

**Платформа находится в правильной точке для своего возраста (9 дней).** Архитектурные решения зрелые, домены чисто разделены, hardening sprint реально закрыл pipeline-correctness с тестами, BusinessTruth — образец того, как пилить новый модуль. Долга на удивление мало для такой скорости.

**Единственный действительно срочный кластер проблем — security и operational hygiene.** P0 (M-001 ротация ключей, M-002 закрытие портов) можно сделать за час и нужно сделать сегодня. Остальные P1 (M-003..M-011) — за неделю. После этого можно начинать multi-tenant спринт с чистой совестью.

**Что нельзя делать:** начинать multi-tenant сейчас, без закрытия M-001..M-008 и M-007 (`tenant_id` миграция). Если позвать второго платящего клиента в текущем состоянии — это будет немедленный security incident, потеря OAuth токенов или Anthropic-биллинга, и неприятный звонок.

**Quick wins (~22-26 часов работы) дают непропорционально большой эффект:** закрывают все P0, большинство P1, поднимают a11y до 60%, срезают 150 KB из главного бандла, чинят performance hot-spots. Это правильный фокус следующих 2-3 дней.
