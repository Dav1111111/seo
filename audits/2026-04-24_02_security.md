**Стандарт:** OWASP ASVS L1 / OWASP Top 10 2021
**Аудитор:** Claude Opus 4.7 (Security review)
**Скоуп:** backend (FastAPI), frontend (Next.js 16), nginx, docker-compose, .env, скоро мульти-тенант.

## TL;DR — Risk Posture

- **Критические незакрытые риски:** **8** (S-001..S-008)
- **High:** **7**, **Medium:** **6**, **Low:** **4**, **Info:** **5**
- **Готовность к мульти-тенанту:** **BLOCKED**
  Авторизации нет вообще. API-эндпоинты типа `/sites/{id}/...`, `/reviews/{id}`, `/reports/{id}`, `/priorities/sites/{id}`, `/intent/sites/{id}/...` принимают любой UUID без проверки владельца. На single-tenant инсталляции это работает по принципу «один владелец, всё его». На мульти-тенанте даст полную утечку чужих данных (IDOR + горизонтальная эскалация).
- **Самая горящая дыра:** Реальные production-ключи (Anthropic, Yandex OAuth, OAuth client secret) лежат в `/Users/davidgevorgan/Desktop/yandex-growth-tower/.env`, файл не в git, но в проде он же читается docker-compose'ом. Любой, кто получит шелл на VPS, заберёт всё. Дополнительно — `ADMIN_API_KEY=admin_dev_secret_2026` хардкодом вшит в `docker-compose.yml:93`, файл закоммичен в git.

Платформа в целом аккуратно построена (ORM, Pydantic, аккуратные валидаторы), но **отсутствует вся аутентификация end-user'ов** и **базовая операционная гигиена секретов**. Перед мульти-тенантом обязательно: (1) встроить аутентификацию пользователей и tenant scoping, (2) ротировать все ключи, (3) добавить rate limiting, (4) поднять nginx до HTTPS + security headers, (5) переехать на Docker secrets / Vault.

## Threat Model (короткая)

**Кто атакует:**
- T1. Случайный сканер портов (Shodan, internet-wide scans). Сейчас порт 80 открыт, Postgres 5432 и Redis 6379 проброшены в `0.0.0.0` через docker-compose.
- T2. Целевой атакующий, узнавший VPS-домен `c6b32ea267b.vps.myjino.ru` (поддомен в `main.py:25`, утечёт через CORS/HTTP headers / любой публичный пост). Цель — снять данные клиентов / Anthropic-кошелёк / OAuth-токен Яндекса.
- T3. Конкурент / "сосед-клиент" в мульти-тенант-фазе. Узнаёт чужой `site_id` и читает чужие KPI, конкурентов, BusinessTruth, отчёты.
- T4. SEO-спамер, который встраивает prompt-injection в текст своей страницы или в поисковый запрос, попадающий в Webmaster API → в Claude. Цель — заставить агента написать рекомендации в свою пользу или вытянуть system prompt.
- T5. Инсайдер с доступом к VPS. Цель — выгрузка БД (Postgres-пароль `devpassword`, порт 5432 наружу).

**Что они могут получить (assets):**
- A1. Ключи третьих сервисов: Anthropic API key (~$10/мес лимит, но привязан к карте), Yandex OAuth token (доступ к Webmaster + Metrica всех хостов владельца), Yandex OAuth client_id/secret (полный приложенческий креденшл).
- A2. Содержимое БД: query strings из Webmaster (PII конечных пользователей сайта-клиента), список конкурентов клиентов, BusinessTruth-понимание (бизнес-стратегия), recommendation_id с before/after-копирайтом.
- A3. Деньги. `/admin-proxy/*` без rate-limit ⇒ массовые dispatch'и Celery-тасков ⇒ Anthropic-биллинг.
- A4. Manipulation. SEO-спам через query strings → LLM-агент вернёт хостайле-рекомендации владельцу.

**Trust boundary:**
- Browser → nginx:80 (plain HTTP, нет TLS) → Next.js:3000 / FastAPI:8000.
- FastAPI → Postgres / Redis (внутренняя сеть Docker).
- Backend → Anthropic (через Cloudflare Worker proxy) / Yandex API.

Сейчас нет ни одной строки кода, которая бы разделяла «внешний пользователь» и «владелец платформы». Всё — single-trust.

#### S-001 — Отсутствие любой аутентификации на пользовательских API

  - `backend/app/api/v1/priority.py:35,54` `GET /priorities/sites/{site_id}`, `weekly-plan`

  - `backend/app/api/v1/intent.py:41-216` все intent-эндпоинты
  - `backend/app/api/v1/collectors.py:22-99` все триггеры коллекторов и пайплайнов
  - `backend/app/api/v1/queries.py:31-...`, `backend/app/api/v1/tasks.py:23-...`, `backend/app/api/v1/fingerprint.py:24-...`
- **CWE / OWASP:** CWE-306 Missing Authentication for Critical Function / **A01:2021 Broken Access Control** + **A07:2021 Identification & Authentication Failures**
- **Сценарий атаки:**
  1. Атакующий находит VPS-домен `c6b32ea267b.vps.myjino.ru` (поддомен явно записан в `backend/app/main.py:25`, можно вытащить через любой ответ FastAPI с CORS-заголовком, или через DNS bruteforce).
  2. `curl http://c6b32ea267b.vps.myjino.ru/api/v1/sites` — получает список всех сайтов с `tenant_id`, `domain`, `yandex_webmaster_host_id`, `yandex_metrica_counter_id`, `display_name`, `created_at`. **Token не нужен**.
  3. Для каждого `site_id` дёргает `/dashboard`, `/queries`, `/issues`, `/reports/sites/{id}/latest`, `/priorities/sites/{id}` и собирает всю SEO-аналитику чужого бизнеса.
  4. `PATCH /sites/{id}/issues/{issue_id}` со `status: resolved` — затирает алерты владельца.
  5. Через `PATCH /sites/{id}` (`backend/app/api/v1/sites.py:53`) — может перезаписать `yandex_oauth_token`, `yandex_metrica_counter_id`, `is_active=false`. Полный hijack профиля сайта.
- **Impact:** на single-tenant сейчас — утечка всей аналитики двух доменов владельца + возможность вандализма. На мульти-тенант — полная горизонтальная эскалация: каждый клиент видит всех. Регуляторное нарушение 152-ФЗ если query strings содержат персональные данные.
- **Fix:**
  - Срочный (single-tenant): закрыть `/api/v1/*` через nginx Basic Auth ИЛИ глобальный `Depends(_require_admin)` на `v1_router`. Это не auth, это хотя бы дверь.
  - Перед мульти-тенантом: сессионная аутентификация (cookie+CSRF либо Bearer JWT). Вся защита через FastAPI dependency, который выдаёт `current_tenant_id`. Всем ORM-запросам, где есть `site_id`, добавить `where(Site.tenant_id == current_tenant_id)`.

#### S-002 — IDOR на всех `/sites/{site_id}/*` маршрутах: проверки владельца нет
- **Где:** все queries, использующие `site_id` из URL, не сверяют его с tenant пользователя. Примеры:
  - `backend/app/api/v1/sites.py:44` — `select(Site).where(Site.id == site_id)` без `tenant_id`
  - `backend/app/api/v1/dashboard.py:38-44` — `where(DailyMetric.site_id == site_id)` без tenant join
  - `backend/app/api/v1/admin_ops.py:53,170,266,308` — admin-endpoints проверяют только `Site.id`, но между tenant'ами нет ограничения
  - `backend/app/api/v1/report.py:32,40` — `GET /reports/{report_id}` принимает любой UUID, лежит ли он в твоём тенанте — не проверяется
  - `backend/app/api/v1/review.py:80-119` — `GET /reviews/{review_id}` и `GET /pages/{page_id}/reviews/latest` — то же
- **CWE / OWASP:** CWE-639 Authorization Bypass Through User-Controlled Key / **A01:2021 Broken Access Control**
- **Сценарий атаки (мульти-тенант):**
  1. Tenant A авторизован, у него `site_id = aaaa-...`. Tenant B имеет `site_id = bbbb-...`.
  2. Tenant A в браузере отправляет `GET /api/v1/sites/bbbb-.../dashboard`.
  3. Backend ничего не проверяет — Tenant A видит дашборд Tenant B.
  4. То же для `/reports/{report_id}` — UUID хорошо угадывать сложно, но он утекает в URL, в логах nginx, в access-log'ах VPS. Один компрометированный logfile = доступ ко всем отчётам.
- **Impact:** в мульти-тенанте — полная утечка чужих данных. SaaS-критичный риск.
- **Fix:**
  1. `Depends(get_current_site_for_user)` — единственная точка, где смотрят URL `site_id`, проверяют принадлежность tenant'у, возвращают объект.
  2. Все `select()` принимают объект, не raw UUID. Защита на уровне sessions: `session.execute(select(Site).where(Site.id == site_id, Site.tenant_id == ctx.tenant_id))`.
  3. Тесты: для каждого роута — позитивный + негативный (другой tenant → 404 или 403).

#### S-004 — Yandex OAuth tokens хранятся в БД в открытом виде
- **Где:**
  - `backend/app/models/site.py:19` — `yandex_oauth_token: Mapped[str | None] = mapped_column(Text)  # encrypted at rest`
  - Комментарий обманывает: ни в одном файле проекта нет применения шифрования. `grep -r "encrypt\|decrypt\|fernet" backend/app/` ничего не находит. `ENCRYPTION_KEY` объявлен в `config.py:36`, но не используется.
  - `backend/app/collectors/tasks.py:137,211,341` — токен читается напрямую: `oauth_token=site.yandex_oauth_token or settings.YANDEX_OAUTH_TOKEN`.
  - `backend/app/schemas/site.py:12,29` — `yandex_oauth_token: str | None` принимается plaintext'ом в `SiteCreate` и `SiteUpdate`.
- **CWE / OWASP:** CWE-312 Cleartext Storage of Sensitive Information / **A02:2021 Cryptographic Failures**
- **Сценарий атаки:**
  1. SQL-дамп Postgres (proxied port 5432, slabый пароль `devpassword`) → `SELECT yandex_oauth_token FROM sites;` → у атакующего полный доступ к Webmaster + Metrica всех клиентов.
  2. Через `pg_dump`, через бэкап-снапшот VPS, через `docker-compose logs` (если SQLAlchemy `echo=True` — а в `database.py:9` `echo=settings.APP_ENV == "development"`, то есть в dev-режиме все запросы пишут токены в логи).
- **Impact:** утечка одного бэкапа БД = доступ ко всем Webmaster-аккаунтам клиентов. У OAuth-токена Яндекса нет короткого TTL, ротация требует ручного вмешательства владельца.
- **Fix:**
  - Завести `ENCRYPTION_KEY` (32 байта, base64) и шифровать `yandex_oauth_token` через `cryptography.fernet` (библиотека уже в pyproject.toml: `cryptography>=44.0.0`).
  - Создать helper `app/services/secrets.py`: `encrypt_token(plaintext)` / `decrypt_token(ciphertext)`. Применить в migration: расшифровать существующие токены (для текущих 1-2 сайтов), зашифровать обратно.
  - `SiteCreate`/`SiteUpdate` принимают plaintext, но шифруют перед `db.add()`.
  - Включить TLS на Postgres-соединение (`sslmode=require` в DATABASE_URL).

#### S-005 — Postgres и Redis проброшены на 0.0.0.0 без аутентификации
- **Где:**
  - `docker-compose.yml:11-12` — `db.ports: ["5432:5432"]` (Postgres с паролем `devpassword`)
  - `docker-compose.yml:22-23` — `redis.ports: ["6379:6379"]` (Redis вообще без пароля!)
- **CWE / OWASP:** CWE-284 Improper Access Control / **A05:2021 Security Misconfiguration**
- **Сценарий атаки:**
  1. Shodan / Censys сканирует диапазоны IP myjino.ru. Видит VPS с открытым 5432 и 6379.
  2. `redis-cli -h c6b32ea267b.vps.myjino.ru` → подключается без аутентификации → может подменять Celery-таски: `LPUSH celery '{"task":"crawl_site","args":["any-uuid"]}'`. Через `CONFIG SET dir /var/spool/cron/crontabs` + `CONFIG SET dbfilename root` + `SAVE` исторически возможен RCE.
  3. `psql -h c6b32ea267b.vps.myjino.ru -U tower -d growthtower` с паролем `devpassword` (видим в .env / в репо или брутфорсится за минуты словарём) → весь дамп.
- **Impact:** полная компрометация. Redis без пароля исторически дал тысячи кейсов RCE через SAVE/CONFIG.
- **Fix:**
  - Удалить `ports:` блоки у db и redis в docker-compose.yml. Они уже доступны контейнерам через имя сервиса (`db:5432`, `redis:6379`) — внешний проброс не нужен.
  - Если порт нужен для миграций/админки — пробросить только на `127.0.0.1:5432:5432` и ходить через `ssh -L`.
  - Включить `requirepass` в Redis (через `command: redis-server --requirepass ...`), пароль в env.
  - Сменить пароль Postgres на длинный, переехать с `devpassword`.

#### S-006 — CORS `allow_credentials=True` + `allow_methods=["*"]` + `allow_headers=["*"]` + statically-listed origins
- **Где:** `backend/app/main.py:32-38`
  ```python
  app.add_middleware(
      CORSMiddleware,
      allow_origins=_allowed_origins,  # localhost + c6b32ea267b.vps.myjino.ru + CORS_ORIGINS env
      allow_credentials=True,
      allow_methods=["*"],
      allow_headers=["*"],
  )
  ```
- **CWE / OWASP:** CWE-942 Permissive Cross-domain Policy with Untrusted Domains / **A05:2021 Security Misconfiguration**
- **Сценарий атаки:**
  1. Эндпоинты не требуют auth (см. S-001), значит CORS-атака пока неактуальна — кому красть, если данные и так публичны? Но как только добавятся cookies/sessions:
  2. Origin `http://c6b32ea267b.vps.myjino.ru` (HTTP, не HTTPS) — атакующий в Wi-Fi-сети жертвы или через DNS spoofing подсовывает свой response, читает cookies.
  3. `CORS_ORIGINS` env читается из произвольной строки и попадает в whitelist — конфигурационная мина: `CORS_ORIGINS=https://attacker.tld` и всё открыто.
- **Impact:** в текущем виде маловажно (нет sessions). Перед добавлением auth — критично.
- **Fix:**
  - Заменить `allow_methods=["*"]` на конкретный список: `["GET","POST","PATCH","PUT","DELETE","OPTIONS"]`.
  - Заменить `allow_headers=["*"]` на `["Content-Type","Authorization","X-Admin-Key","X-Requested-With"]`.
  - Запретить любые HTTP-origins (`http://...`) в проде — только HTTPS.
  - `CORS_ORIGINS` валидировать (regex для https + конкретный домен).
  - Поднять nginx до TLS (Let's Encrypt) и убрать `http://c6b32ea267b.vps.myjino.ru` из whitelist.

#### S-007 — nginx без TLS, без security headers, без rate limiting
- **Где:** `nginx/nginx.conf` весь файл (38 строк)
- **CWE / OWASP:** CWE-319 Cleartext Transmission, CWE-693 Protection Mechanism Failure / **A05:2021** + **A02:2021**
- **Сценарий атаки:**
  1. Весь трафик `http://c6b32ea267b.vps.myjino.ru/...` идёт plain HTTP. Любой MITM (Wi-Fi, ISP, корпоративный proxy) видит API-payload, BusinessTruth, заголовки.
  2. Нет CSP — XSS любого формата работает (если будет).
  3. Нет `X-Frame-Options` или `frame-ancestors` — clickjacking тривиален.
  4. Нет `Strict-Transport-Security` — даже после ввода TLS пользователь может уйти на HTTP downgrade.
  5. Нет rate limit — атакующий может отправить 10 000 RPS на `/api/v1/admin/sites/{id}/pipeline/full` (если знает ключ), или на не-auth эндпоинт `/api/v1/sites` для энumeration.
  6. `server_tokens` не выключен — отдаётся версия nginx (фингерпринт для CVE-сканера).
  7. `/docs` (Swagger) открыт публично (`nginx.conf:30`) — карта всего API на блюдечке для атакующего.
- **Impact:** базовая web hygiene не выполнена. Перед запуском мульти-тенанта это must-have.
- **Fix (nginx.conf):**
  ```nginx
  server_tokens off;

  # Rate limit zones
  limit_req_zone $binary_remote_addr zone=api:10m rate=10r/s;
  limit_req_zone $binary_remote_addr zone=admin:10m rate=2r/s;
  limit_req_zone $binary_remote_addr zone=chat:10m rate=1r/s;

  server {
      listen 443 ssl http2;
      ssl_certificate     /etc/letsencrypt/live/.../fullchain.pem;
      ssl_certificate_key /etc/letsencrypt/live/.../privkey.pem;

      add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;
      add_header X-Content-Type-Options "nosniff" always;
      add_header X-Frame-Options "DENY" always;
      add_header Referrer-Policy "strict-origin-when-cross-origin" always;
      add_header Content-Security-Policy "default-src 'self'; script-src 'self' 'unsafe-inline'; style-src 'self' 'unsafe-inline'; img-src 'self' data:; connect-src 'self'" always;

      location /api/ { limit_req zone=api burst=20 nodelay; ... }
      location /api/v1/admin/ { limit_req zone=admin burst=5 nodelay; ... }
      location = /docs        { return 404; }   # or basic-auth
      location = /openapi.json { return 404; }
  }
  ```

## Что сделано хорошо

1. **ORM везде, raw SQL только параметризованный.** `text("SELECT pg_try_advisory_lock(:k)")` с bind-параметрами — корректный паттерн. SQL injection нет нигде в проде.
2. **Pydantic v2 валидаторы на входе.** `SiteCreate.validate_domain`, `Field(pattern=...)`, `min_length`, `max_length` — везде.
3. **`yandex_oauth_token` явно исключён из API response** (`schemas/site.py:48`). Это правильное намерение, хотя само хранение plaintext'ом сводит этот win на нет (см. S-004).
4. **Admin-эндпоинты вынесены в отдельный prefix `/admin`** + `_require_admin` через FastAPI `dependencies=[Depends(...)]`. Архитектурно правильно.
5. **Frontend admin-proxy скрывает X-Admin-Key от браузера** — хороший паттерн. Просто его нужно расширить per-user auth (S-107).
6. **В review LLM-prompts уже есть антипромпт-инжекшн guard** (`prompts.py:38`). Образец для других агентов (S-104).
7. **Empty `ADMIN_API_KEY` = всё закрыто** (`admin_ops.py:32`, `admin_demand_map.py:50`). Fail-safe by default — отлично.
8. **`get_db()` правильно делает rollback при exception** (`database.py:46-49`). Транзакционная безопасность есть.
9. **Pydantic `tool_use` для LLM** в `chat_agent.py`, `business_understanding.py`, `review/llm/prompts.py` — устойчивее JSON-парсинга, меньше шансов на injection через JSON malformation.
10. **Celery tasks через `apply_async` с `countdown`** — нет race conditions на старте.

## Приложение B — Внешние зависимости с поверхностью атаки

- **Cloudflare Worker proxy для Anthropic** (`config.py:22` — `ANTHROPIC_BASE_URL`). Не ясно, есть ли там auth: если этот proxy «открытый», любой кто узнает URL может слать запросы за счёт нашего ключа. **Действие:** проверить наличие auth на Worker'е (Cloudflare Access / shared secret в header).
- **Yandex Webmaster / Metrica** — токен один на всех клиентов сейчас (settings.YANDEX_OAUTH_TOKEN fallback в `collectors/tasks.py:137`). На мульти-тенант нужно или per-tenant OAuth flow, или явное предупреждение клиенту что его данные ходят через общий токен.
- **Yandex Cloud Search API** — ключ `YANDEX_SEARCH_API_KEY` (config.py:51), не в .env, видимо ещё не настроен. При настройке — те же риски (S-003).
